Who need it?

  • Intended mainly for use by non-programmers
  • ... especially web authors and designers
  • It's not all things for all people
    • it's not intended for large programming tasks
    • ...but it's great for transforming documents
    • it's not intended for high-volume websites
    • ...but it's great for workgroups and individuals
    • ...especially webmasters

What do we use it for?

  • Permanent web history
    where / when / site structure
  • Website maintenance
    link-checking / style consistency
  • Workflow
    a complete web application development environment
    all of RSV's forms are on the web -- 6 man-weeks.
  • Automated documentation
    pretty-printing / comment extraction / indexing
  • Prototyping
    interactive websites / networked appliances

Document-processing goals

What the Userswant:

 

  • Documents have to be maintainable
    • Clean interface between documents and code
    • XML representation for all persistent data
  • Easy to use by authors, not programmers
    • Code and data have the same syntax (SGML)
  • Use browser for user interface
  • Adhere to standards wherever possible
    • SGML, HTML, SML...

Implementation goals

What the developers want:

 

  • The system has to be maintainable
    • All source code online and browsable
    • Much documentation derived from the code
    • Server as self-documenting as possible
  • Code has to be portable
    • Portable languages (Java, Perl)
    • Avoid user interface code
    • Avoid binary file formats
  • Adhere to standards wherever possible
    • Standard representation: DOM
    • Standard protocol: HTTP
    • Standard syntax: HTML, XML, SGML

What it's not

  • Not a CGI scripting language
    • Perl, Java, Tcl, etc.
  • Not a scripting language embedded in web pages
    • asp, jsp, JavaScript, ...
  • Not servlets or server-side includes
  • Not a style-sheet language (though this comes close)
    • DSSSL, cascading style sheets, ...
  • Not (just) an extended HTML
    • MetaHTML, php, ...
  • Not a high-volume web server
    • We recommend Apache for that.

What it is

  • Smoothly integrated into SGML documents
    • no new syntax to learn
    • easily handled by SGML-aware editors
  • Easily integrated into document workflow
  • A seamless part of the Web
  • Designed for cut-and-paste document development
  • A personal tool for operating on documents

What's different?

  • XML syntax for actions
    • no CGI's or embedded Basic/Perl/Java/etc. code
    • easily edited with standard tools
    • ``programs'' can be processed as documents
    • programmers not required for most work
      -- Technical writers can do it.
  • Generic SGML as well as XML and HTML
  • Optional separation of behavior and content
    • actions can be separate (tagset) or embedded
      The syntax is the same in either case
    • documents can be processed with different sets of actions for the same tags (tagsets)
    • Dynamically-loaded handlers
  • Entities (&foo;) used as variables.
    • can be written as well as inserted
