  • Fill out and process forms

  • Create and sign legal documents
  • Edit and publish manuscripts
  • Memos: send, recieve, copy, file, ...
  • Letters, bills, junk mail, magazines, postit notes, ...

Processing:

  • signing: approving, authorizing, ...
  • storing: indexing, filing, stacking, boxing, ...
  • retrieving: searching, skimming, ...
  • excerpting, annotating, summarizing, ...
  • presenting, performing, parodying, ...

The Problem

It all goes back to the nature of documents on computers:
  • Documents and software have different languages
    • usually in separate files
  • Developing applications requires software
    • There's never a programmer around when you need one.

Also,
  • Applications and documents change
    • programs must be maintained
    • documents must be updated
    • it's hard to keep them in sync.

How we do it

(OK -- this is the good stuff!)

  • Documents have embedded processing instructions
    • Everything needed for a form is in one file
  • But only one language (XML)
    • In other words, we simply define actions for some of the tags.
  • The meanings of tags can change
    • So a document can be used in several ways
    • One document (a tagset) can define tags for another document.

Here's how we build a form:

  • HTML and processing instructions 
    <form action='&url;>    
  <set name=vendor><get name=PR:vendor></set>
    <!-- ... -->
  Vendor: <input name=vendor value=&vendor;><br>
  Phone: <input name=vphone value=&vphone;>
  <process>
    <if><get name=approved> 
       <then> <send-email dst=purchasing>
		 <!-- ... --> 
		 Vendor: &FORM:vendor;
                 Phone: &FORM:vphone;.
	      </send-email>
       </then>
    </if>       
  </process>
</form>

A Framework for Infoware

The PIA is what we sometimes call a ``framework:''

  • A re-useable design for a class of applications.
Agents:
 
  • Building applications is really easy
  • ...because active documents are easy to write
 
Infoware 
Agency:
 
  • The software base is stable
  • ...and shared by all applications
 
Software  

More about Frameworks:

A framework consists of:
  1. A shared software base
    • stable, so most applications don't need to change it
    • open, so that it can be examined and extended
  2. Lots of documentation
    • User's manual, tutorials, etc.
    • Internals, maintenance documents, etc.
  3. Working sample applications (agents)
    • Meant to be extended and modified
    • The designer starts from something that already works

The Open-Source Community

  • The typical infoware application is mostly information
    • perhaps 80% documents, 20% processing
  • In the PIA, most of that processing can be done in the documents using tags.
  • An application may need one or two task-specific tags
    • e.g. digital signatures in the form agent
  • But with an open-source community, someone probably had the same need, and filled it.
