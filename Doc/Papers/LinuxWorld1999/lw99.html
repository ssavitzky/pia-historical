Documents and the Web

Appliances

The future of computing is networked appliances
  • (NOA = Networked Office Appliance)
  • Web-enabled -- used mainly or only through a browser.
  • Mostly special-purpose
  • May be clients, servers, or (ta-da!) document processors

Appliances on the Web

An appliance contains a thin server.
  • Used through its Web pages
  • Pages are generated by ``agents''
    • from ``active documents''
    • and (optional) custom code
  • Agents run on ``Agency'' infrastructure
    • Processes HTTP requests
    • Implements XML standard
    • Runs agents
  • Examples:
    • WebPrinter
    • Photo Album
    • Form server

Document Processing

InfoWare, not software

  • Information is more important than either interface or processing
    • above all, preserve the information!
    • If information is accessible, processing is straightforward.
  • Informal knowledge targeted at an individual or group
    • mostly documents, not databases
    • hierarchical, not tabular
    • every group has its procedures
  • Maintainability over the long run is key
    • No proprietary formats
    • No secret algorithms

What Are Documents?

  • A document is a tree-structured collection of text
  • Document = content (text) + structure (markup tags)
  • Documents may have
    • multiple versions
    • multiple presentations (styles / views)
    • multiple representations (file / parse tree / ...)
    • multiple ``behaviors''

A Document is not an Object

  • Documents separate content from behavior
    Objects combine content and behavior
  • Therefore, documents...
    • are portable
    • can be processed in many ways
      [formatted, indexed, translated, ...]
    • are robust (unlike serialized Java objects)

Why Process Them?

  • The ``traditional'' operations
    • Creating / editing
    • Viewing
    • Printing (typesetting)
    • Indexing
    • Searching
  • But wait! There's more!
    • Reformatting (applying styles)
    • Managing collections (e.g. web sites)
    • Translating / converting /compiling
    • Extracting information
    • Filtering
    • Form handling
    • Workflow / groupware

What Can Our System Do?

  • Short answer -- all of the above
  • ... to any SGML document (including HTML and XML)
  • ... any other document, as long as someone writes a parser
  • On-the-fly or offline (command line)
  • As server, client, or proxy

  • ``A spreadsheet for documents''

What do we use it for?

  • Permanent web history
    where / when / site structure
  • Website maintenence
    link-checking / style consistency /
  • Form Handling
    a complete web application development environment
    all of RSV's forms are on the web -- 6 man-weeks.
  • Automated documentation
    pretty-printing / comment extraction / indexing
  • Prototyping
    interactive websites / networked appliances

Who can use it?

  • Intended mainly for use by non-programmers
  • ... especially web authors and designers
  • As we'll see later
    • it's not intended for large programming tasks
    • but it's great for transforming documents

What it's not

  • Not a CGI scripting language
    • Perl, Java, Tcl, etc.
  • Not a scripting language embedded in web pages
    • asp, jsp, JavaScript, ...
  • Not servlets or server-side includes
  • Not a style-sheet language (though this comes close)
    • DSSSL, cascading style sheets, ...
  • Not (just) an extended HTML
    • MetaHTML, php, ...
  • Not a high-volume web server
    • We recommend Apache for that.

What it is

  • Smoothly integrated into SGML documents
    • no new syntax to learn
    • easily handled by SGML-aware editors
  • Easily integrated into document workflow
  • A seamless part of the Web
  • Designed for cut-and-paste document development
  • A personal tool for operating on documents

What's different?

  • XML syntax for actions
    • no CGI's or embedded Basic/Perl/Java/etc. code
    • easily edited with standard tools
    • ``programs'' can be processed as documents
    • programmers not required for most work
      -- Technical writers can do it.
  • Generic SGML as well as XML and HTML
  • Optional separation of behavior and content
    • actions can be separate (tagset) or embedded
      The syntax is the same in either case
    • documents can be processed with different sets of actions for the same tags (tagsets)
  • Dynamically-loaded handlers

Document-Oriented Computing

 

  • A whole new way of looking at computation
    • variables contain documents
    • content and behavior are separate
  • A new location for computation
    • documents get passed between nodes
    • not just clients and servers
    • peers and proxies as well

A Simple Example

  • Input: a document 
           <slide> <h2>Document Processing</h2>
          <ul>
             <li> Input: a document
          </ul>
       </slide>
  • ... processing using a ``tagset''
    • A ``spreadsheet'' for documents
    • Describes what to do with each tag
  • process -tagset slides lw99.html > lw99.slides.html
  • Output: another document (this slide).

What's Going On

  • A document's format is specified by a
    Document Type Definition (DTD)
  • ... which is another document!
  • ... which we can represent in XML.
  • ... and process

  • A ``tagset'' is a document that combines
    a DTD with formatting commands.
  • ... also represented in XML.
  • ... extended with standard programming constructs (XXML)
  • Any document may contain ``active'' constructs
  • ... resulting in ``active documents'' in XXML and XHTML

Another Example

  • Input: a document tagset specification 
      <tagset name=slides parent=HTML include=xxml recursive> ...
  <define element=slide parent=body> <action> ...
  <define entity=topBg><value>lightblue</define>...
  • ... processing using a different tagset
  • process -tagset tsdoc slides.ts > slides.html
  • Output: documentation.

What's Going On?

  • Documents contain only content and markup (structure)
  • Behavior is associated with tags by a tagset
  • The same document can be processed in different ways with different tagsets.
    -- think of them as similar to styles

  • One way to look at it:
    • we're abusing a text formatter
    • ... to make a programming language

What's Really Going On?

 

A Document-Processing Engine for the Web
  • The Web is made of documents
  • ``In the future, the only API's that matter will be the API's of web sites''
    -- Tim O'Reilly
  • What we have is a general-purpose document processor
  • designed specifically for web applications.
    • Can operate as client, server, proxy, or any combination
    • Can process other servers' documents
      ... examples further on
  • Serves as the basis for
    • personal web assistants
    • easy-to-administer servers
    • web-based information appliances
Architecture and Implementation

Architecture and Implementation

  PIA System Architecture  
     

Document-processing goals

What the user sees.
  • Documents have to be maintainable
    • Clean interface between documents and code
    • XML representation for all persistent data
  • Easy to use by authors, not programmers
    • Code and data have the same syntax (SGML)
  • Use browser for user interface
  • Adhere to standards wherever possible
    • SGML, HTML, SML...

Implementation goals

What the developer sees.
  • The system has to be maintainable
    • All source code online and browsable
    • Much documentation derived from the code
    • Server as self-documenting as possible
  • Code has to be portable
    • Portable languages (Java, Perl)
    • Avoid user interface code
    • Avoid binary file formats
  • Adhere to standards wherever possible
    • Standard representation: DOM
    • Standard protocol: HTTP
    • Standard syntax: HTML, XML, SGML

Agency Architecture Layers

  • Server Front End
    • Handles HTTP requests
    • Dispatches each request to ``agent'' for response
    • Allows agents to ``act on'' documents as they flow through
  • Agent handlers
    • Manage a group of pages
    • May have special-purpose code.
  • Document Processor for active pages
    • Interprets markup in active documents
    • Rarely needs to be changed
    • Recently re-written using DOM
  • ``Action'' handlers for tags
    • Written in XML. Java, or C
    • Loaded as needed at run-time
    • Organized into ``tagsets''

PIA's Document Processor

  • Simple
    • Main loop fits on a single page.
    • Easy to maintain and describe.
  • Versatile
    • stream, string and parse tree input and output
    • can build parse trees while processing
  • Portable
    • based on W3C's Document Object Model
    • C or C++ version would be feasible
    • An earlier version was written in Perl

The Implementation

  • General Technique:
    • Processor gets input from a parse tree traverser
    • Sends output to a parse tree constructor
  • Algorithm:
    • Convert input to a parse tree.
    • Each ``element'' has a parent Node and content NodeList.
    • Parser associates an action with each node
    • Traverse the parse tree depth first
    • Actions pass result nodes to an output tree constructor
  • Advantages:
    • Basic algorithm is very simple
    • All table lookup is done once, in the parser
    • Saving parse trees in server RAM gives a major speedup

Primitive Actions

(all are active SGML elements)
  • <define>
  • <get>, <set>
  • <if> - <then> - <else-if> - <else>
  • <test>, <logical>
  • <repeat>
  • <select>
  • <expand>, <protect>, <hide>
  • <text>, <numeric>
  • <subst>

I/O and Data Conversion

(additional primitives)
  • <parse>
    converts raw text to marked-up XML.
  • <include>, <output>, <connect>, <status>
    I/O, including both files and URL's.
  • <DOCUMENT>, <HEADERS>, <URL>, <QUERY>, ...
    data structure representation and conversion
  • <debug>

There's more...

(but not much more:)

  • There are a few elements used inside the primitives
  • ... most primitives are modified by attributes,
  • ... but the basic language is still fairly small.

  • New tags can be easily defined
    • usually in terms of old ones
    • can connect to Java ``handler''

  • SGML &entities; are used as variables.

For example...

<define element=factorial recursive>
  <doc>the Factorial function</doc>
  <action>
    <if><test zero>&content;</test>
      <then>1</then>
      <else><numeric product>
              &content;
              <factorial><numeric difference>&content; 1
                         </numeric>
              </factorial>
           </numeric>
      </else>
    </if>
  </action>
</define>
It's not really meant for numeric processing!

A better example...

It's really meant for web pages. 
<html>
  <head>
    <title> Hello world!<title>
  </head>
  <body>
    <header> a classic example </header>
    <hr>
    Hello world!
    <footer cvsid="$ Id... $">
  </body>
</html>
...which becomes this page on the server. (local copy)

Tags and actions are separate

You can define a new set of tags. 
<html>
  <tete>
    <title> Bonjour, tout le mond!<title>
  </tete>
  <corps>
    <bienvenue> une example classique </bienvenue>
    Bonjour, toute le mond!
    <pied cvsid="$ Id... $">
  </corps>
</html>
as well as redefining the old ones.
The language is easily localized or abbreviated.

So the first example...

...wasn't really so bad after all! 

<element name=fact>
  <doc>the Factorial function, again.
  <defined-as>
    <if><zerop>&content; <then> 1
      <else><product> &content;
                      <fact><diff>&content; 1
</element>
(Especially if you leave out end tags wherever possible.)

Open Source

Why Open Source for Us?

  • Software is a cost for Ricoh
    • We mostly sell hardware. We need software to make it useful, but can't sell it separately.
    • Eric Raymond's ``widget frosting'' business model
  • We want the PIA to become a de facto standard.
    • It's not nearly as useful if it isn't widely used
    • Needs a large user/developer community
  • We use lots of open-source software
    • and want to give something back to the community
  • It's best for the customers
    • We target Small Office/Home Office applications
    • Small businesses don't have lots of money for software

Why Open Source for Developers?

  • Solid, peer-reviewed software
  • Active developer community
    • Mutual assistance
    • Fast bug fixes
    • Rapid evolution/improvement
  • No orphan products
    • someone can always take over the project

Open Standards for Document Processing

  • We hope to do for document processing
  • ... what HTML did for document representation
  • ... and HTTP did for document communication

  • Community-developed standards are best
